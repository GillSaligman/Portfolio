7-DOF Robotic Arm

Modular, full-stack mechatronic system designed for autonomous surface exploration

Overview

Inspired by NASA’s RASC‑AL competition, my senior design team and I created a prototype lunar maintenance rover featuring a modular manipulator, custom chassis, and distributed control—built for adaptability, repairability, and upgradeability.

This was a college prototype platform to test modular arm control and system integration—not a sealed production system.

Project Highlights

I served as Team Leader and was responsible for all mechanical design, electronics, software, prototyping, testing, and assembly—as opposed to focusing only on writing documentation.

While RASC‑AL prompted modular swarm ideas, we focused on a single chassis with an advanced modular arm—choosing modularity for easier servicing, upgrades, and upgrade paths. We then built custom arm components and used ROS2-inspired communication architecture.

Mechanical Design

Modular joint layout minimized singularities. Performed torque & force analysis with our target payload, then designed, 3D‑printed (PLA/ABS/PETG) each joint housing around 1×1 aluminum tubing and rapid-prototyped individual joints sequentially.

  • Designed custom brackets, gears, mounts, and joints.
  • Considered, but ultimately sidelined a harmonic drive sub‑assembly due to complexity and timeline.
  • Built 4 joints + gripper (expected 7); J6 axial joint required down‑gearing potentiometer for ±300° rotation using thrust + radial bearings to maintain hollow wire routing.

Electronics

All joints connect to a single LV/HV power snake—only four wires run up the arm, go through a custom PCB, then split to each ESP‑controller. Chassis electronics use standard hobby components (12/24 V supplies, buck converters, fuses, Raspberry Pi).

We evaluated multiple microcontrollers for analog input, Wi‑Fi support, micropython compatibility, and performance. Chosen ESP8266 D1 Minis run MicroPython (not CircuitPython)—optimizing speed for PID loops. Each joint parses pot‑based position, runs PID, and drives motor independently.

Software & Control

I implemented a distributed control architecture: Controller ↔ PC ↔ Raspberry Pi ↔ joint ESPs. The Pi hosts wifi via hostapd/dnsmasq/Mosquitto; PC connects via SSH to launch ROS2‑style nodes.

  • Three bracket‑based data types: Controller (velocity), Desired (position P... or V...), and Actual (from ESPs).
  • System supports joint velocity control with “V<±1>” prefix, or direct joint positioning with “P”.
  • ESP routines: try Wi‑Fi, parse desired, convert to pot‑angle regression, run PID, publish actual at ~10 Hz.

Prototyping & Assembly

Focused on modular design: each joint was built and validated individually—preventing integration bottlenecks. Software was developed in reverse, from end‑effector back to base.

Started with J6 + gripper—software ignored earlier joints until they were built, avoiding errors caused by missing data slots and enabling incremental testing.

Takeaways & Future Work

This platform is designed for future developers to build upon: perfect for demonstrating PID tuning, adding drivetrain, new manipulators, end‑effectors, etc.

Professors plan to use it for student labs showcasing control tuning; next teams can focus on drivetrain, arm expansion, interchangable tools. Complete documentation will support easy handoffs.
